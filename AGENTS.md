# AGENTS.md

This file provides guidance to Qoder (qoder.com) when working with code in this repository.

## Project Overview

This is an **AstrBot plugin** (`review_my_knowledge`) - a knowledge point review assistant. It provides a quiz system with multiple question types (single fill-in-blank, multi fill-in-blank, true/false, open questions) for subjects like biology, chemistry, and physics.

**Version:** 2.2.1

**Official AstrBot plugin documentation:** https://docs.astrbot.app/dev/star/plugin-new.html

## Architecture

### Core Files

- **`main.py`** - Plugin entry point. Defines `KnowledgePlugin` class registered with `@register("review_my_knowledge", ...)`. Contains commands for review books, mistake review, knowledge point review, search, and stats.
- **`knowledge_system.py`** - The main logic. Contains five major classes:
  - `DatabaseManager` - Async SQLite database management using `aiosqlite`. Handles CRUD for entries, stats, user records, pending questions, and user review stats. Uses field whitelist to prevent SQL injection. **`clear_pending` cleans up old session records; `add_pending` only inserts without cleaning.**
  - `UserLogManager` - File-based user logging system. Tracks per-user total ask/exhibit counts and per-entry ask/exhibit/correct/wrong counts in JSON log files. **Includes automatic log rotation (max 1000 lines).** **Counting timing:** `record_ask` increments `total_ask` and `ask` when questions are displayed; `record_exhibit` increments `total_exhibit` and `exhibit` when content is shown; `record_result` only updates `c`/`w` (correct/wrong), does NOT increment `total_ask` or `ask` to avoid double counting.
  - `AnswerMatcher` - Rule-based answer matching engine with **direct match only** and simple yes/no rules. **Logic true/false rules only apply to true/false question types to prevent false positives.** No fuzzy matching or semantic equivalence.
  - `LLMJudge` - LLM-based answer judgment using AstrBot's configured provider. Includes prompt injection protection and 2-minute timeout safeguards.
  - `KnowledgeSystem` - Orchestrator that ties together database, matcher, LLM judge, and user log manager.

### Supporting Files

- **`Init.py`** - Initialization script. Creates `transfer.py` (with embedded txt-to-entry converter) and `settings.txt`. Detects OS type (Windows/Mac/Linux) and AstrBot installation method (Docker vs local). **Runs outside the AstrBot plugin context** to ensure proper plugin configuration.
- **`transfer.py`** - File conversion tool (generated by `Init.py`). Converts raw txt exam files to plugin entry format. **Runs outside the AstrBot plugin context**.
- **`settings.txt`** - Configuration file storing review book file names, descriptions, and AstrBot installation path. **Located in the plugin directory, managed by Init.py**.
- **`configure.py`** - File transfer script with Docker and local mode support. Reads `settings.txt` and copies files to plugin directory. **Docker mode:** uses `docker cp` command to copy files to the container (container name: `astrbot`). **Local mode:** uses `shutil.copy2` with sudo fallback for permission issues. **Can run standalone** (falls back to standard logging if astrbot is not available). **Runs outside the AstrBot plugin context**.
- **`tool_simple.py`** - Original txt-to-entry format conversion tool (source code, can be deleted after Init.py runs as its content is embedded in Init.py). **This is a test file and is not used by the plugin**.

> **注意：** `Init.py`、`transfer.py`、`configure.py`、`settings.txt` 和 `tool_simple.py` 均在 AstrBot 插件外部运行，用于确保 AstrBot 插件正常配置。它们不属于插件核心代码，不参与插件运行时逻辑。

### Data Model

SQLite database (`knowledge_plugin/knowledge.db`) with 5 tables:
- `entries` - Knowledge entries/questions (id, kb_name, category, subject, question_type, answers as JSON, etc.)
- `stats` - Per-entry access counts (total_ask, total_exhibit). **Incremented when questions are displayed to users.**
- `user_records` - Per-user per-entry correct/wrong counts
- `pending_questions` - Session-based pending questions with expiration. **`clear_pending` cleans up per session; `add_pending` inserts without cleaning.**
- `user_review_stats` - Per-user per-entry review tracking (ask_you, exhibit_you, last_ask_time, last_exhibit_time)

User log files stored in `knowledge_plugin/user_logs/<user_name>.log` (JSON format, one record per line). **Log rotation:** files are automatically truncated to the last line when exceeding 1000 lines.

### Counting Consistency (Database vs Log Files)

**Important:** Both database and log files track the same events, but at different times:

| Event | Database | Log File | When |
|-------|----------|----------|------|
| Display question (错题复习) | `stats.total_ask` +1 via `increment_stat()` | `total_ask` +1, `entries[id].ask` +1 via `record_ask()` | When `/开始复习错题` shows questions |
| Show content (知识点复习) | `stats.total_exhibit` +1 via `increment_stat()` | `total_exhibit` +1, `entries[id].exhibit` +1 via `record_exhibit()` | When `/开始复习知识点` shows knowledge points (pure display, no answering) |
| Submit answer | `user_records.c` or `w` +1 via `record_answer()` | `entries[id].c` or `w` +1 via `record_result()` | When `/作答` submits answers (only for 错题复习) |

**`total_ask` and `ask` are incremented ONLY when questions are displayed, NOT when answers are submitted.** This ensures database and log file counts remain consistent.

### Plugin Configuration

Config keys (from AstrBot config): `llm_judge`, `llm_threshold`, `use_llm_for_explanation`, `default_kb`, `session_timeout`, `max_content_length`, `message_max_length`.

## Commands

| Command | Description | Parameters |
|---------|-------------|------------|
| `/列出复习册` | List all review books from settings.txt | None |
| `/开始复习错题` | Start mistake review with smart prioritization (single question mode) | `<复习册名>` |
| `/作答` | Submit answer(s) (仅用于错题复习) | `<答案>` or `1.答案1;答案2\n2.答案3` |
| `/出示答案` | Show answer without judgment | None |
| `/开始复习知识点` | Start knowledge point review (纯展示，无需作答) | `<复习册名> [数量N]` (N<=10) |
| `/搜索知识点` | Search across all databases | `<关键词>` |
| `/生成解析` | Generate explanation using LLM for the most recently shown entry | None |
| `/我的统计` | View personal stats | `[知识库名]` |
| `/重载复习册` | Re-initialize review system (admin) | None |

### Command Details

**`/列出复习册`** - Reads `[DATABASE]` section from `settings.txt`, displays FILES, SHOWNAMES and DESCRIPTION pairs. The SHOWNAMES field is used as the display name instead of file names.

**`/开始复习错题 <kb_name>`** - Smart question selection (single question mode):
1. Priority 1000: Questions user has never seen (ask_you == 0)
2. Priority 500+: Questions where W > C (more wrong than correct)
3. -200 penalty: Questions asked within last 6 hours
4. Only 1 question is selected per review session. Outputs include `total_ask` (global, incremented at display time) and `ask_you` (personal, shows value before incrementing to match sorting logic).

**`/作答 <answers>`** - Submit answer for the current pending question:
- Single question: `/作答 答案`
- Updates: C+1 (correct) or W+1 (wrong). **`total_ask` and `ask` are NOT incremented here** - they were already incremented when questions were displayed.

**`/出示答案`** - Shows answer for the current pending question without judgment. C/W unchanged. **No count increment (`total_ask` already incremented when question was displayed).**

**`/开始复习知识点 <kb_name> <n>`** - 知识点复习是纯展示性质的，用于用户查看和复习知识点内容，**不支持也不需要使用 `/作答` 命令**。优先展示 `exhibit_you` 值低的条目（按 `exhibit_you` 升序排序，优先展示对用户展示次数少的知识点）。Max 10 items. Updates total_exhibit+1 and exhibit_you+1. **`exhibit_you` displays the value before incrementing to match sorting logic.** Shown entries are added to `pending_questions` with `answered=1` to enable `/生成解析` command usage afterwards.

**Note:** When knowledge point review falls back to showing questions (no pure knowledge points available), questions are also counted with `total_exhibit` and `exhibit_you`, same as knowledge points.

**`/搜索知识点 <content>`** - Searches content, answers, and explanation across ALL databases using LIKE query.

**`/生成解析`** - Generates explanation for the most recently shown entry using LLM. Works after `/作答`, `/出示答案`, or `/开始复习知识点`. Retrieves the latest `answered=1` record from `pending_questions` table. Only generates for entries without existing explanation. If multiple entries were shown (e.g., knowledge review with N>1), only the most recent one is used.

## Entry Format

Entries follow this format:

```
<知识库名>
ID=<unique_id>
CATEGORY=<category>
SUBJECT=<生物|化学|物理|通用>
[单填空](Q)问题内容[答案1/答案2/答案3]
解析: explanation text
```

Multi fill-in-blank format:
```
[多填空](Q)问题___和___[答案1|可选1;答案2|可选2]
```

## Development Notes

- **No build step** - This is a Python plugin, loaded directly by AstrBot.
- **Dependency:** `aiosqlite>=0.19.0` (see `requirements.txt`)
- **Database:** Async SQLite via `aiosqlite`. Connection handling uses `@asynccontextmanager` with auto-commit/rollback. Uses `datetime.now(timezone.utc)` instead of deprecated `datetime.utcnow()`.
- **FTS5 Search Index:** `entries_fts` virtual table for full-text search. **Synchronization rules:**
  - **INSERT/UPDATE**: Use `INSERT OR REPLACE INTO entries_fts(rowid, content, answers, explanation) VALUES ((SELECT rowid FROM entries WHERE id = ?), ...)` to ensure precise rowid mapping.
  - **UPDATE explanation**: Use `UPDATE entries_fts SET explanation = ? WHERE rowid = (SELECT rowid FROM entries WHERE id = ?)` to avoid matching duplicate content.
  - **SELECT**: JOIN with `entries` table via `rowid` to return complete fields (id, kb_name, category, subject, question_type, etc.).
  - **Note**: FTS5 uses `rowid` as the implicit link to the main table. Always use `rowid` for synchronization, never `content` matching.
- **LLM Integration:** Uses `context.get_using_provider()` from AstrBot for LLM calls. Always wrapped with `asyncio.wait_for()` with 120-second timeout. LLM prompts only check for **literal answer matches** - no semantic equivalence or terminology variation.
- **Caching:** In-memory dict cache with `asyncio.Lock` and FIFO eviction (max 128 entries).
- **Safety:** Input sanitization via `sanitize_for_prompt()` to prevent prompt injection. JSON parsing via `safe_json_loads()` with error recovery. Stat field whitelist prevents SQL injection.
- **Input validation:** Count parameter is fixed to 1 for mistake review (single question mode), clamped to [1, 10] for knowledge review.
- **Logging:** All modules use `from astrbot.api import logger` for consistent logging. `configure.py` falls back to standard `logging` module when astrbot is not available.
- **User Logs:** File-based JSON logs stored in `knowledge_plugin/user_logs/<user_name>.log`. Each append creates a new line with full state. Read last line for current state. **Automatic rotation at 1000 lines.** See "Counting Consistency" section above for details on when each counter is incremented.
- **Settings:** `settings.txt` stores review book configuration (FILES, SHOWNAMES, DESCRIPTION) and AstrBot installation path (PATH). The SHOWNAMES field provides display names that replace file names when listing review books.
- **Init Script:** `Init.py` detects OS type (Windows/Mac/Linux) and suggests appropriate copy commands (`copy` for Windows, `cp` for Mac/Linux). The txt-to-entry converter code is embedded in Init.py as fallback.

## AstrBot Plugin Development Reference

### Plugin Structure

```
astrbot_plugin_xxx/
├── main.py              # Entry point (required)
├── metadata.yaml        # Plugin metadata (required)
├── requirements.txt     # Dependencies (optional)
├── logo.png             # Icon, 256x256 recommended (optional)
└── config.yaml          # Plugin config (optional)
```

**Key rule:** Persisted data must be stored under the `data` directory, NOT the plugin directory, to prevent data loss during updates/reinstalls. Use `StarTools.get_data_dir("plugin_name")` to get the correct path.

### Basic Plugin Template

```python
from astrbot.api.event import filter, AstrMessageEvent
from astrbot.api.star import Context, Star, register
from astrbot import logger

@register("plugin_name", "author", "description", "1.0.0", "https://github.com/repo")
class MyPlugin(Star):
    def __init__(self, context: Context):
        super().__init__(context)
        logger.info("Plugin initialized")

    @filter.command("hello")
    async def hello(self, event: AstrMessageEvent):
        '''Command description (shown in help)'''
        yield event.plain_result(f"Hello, {event.get_sender_name()}!")
```

### Command System

- **Basic command:** `@filter.command("name")` - cannot contain spaces
- **With parameters:** AstrBot auto-parses args: `async def cmd(self, event, a: int, b: str)`
- **Command groups:** Use `@filter.command_group("group")` to create a class with nested `@filter.command` methods

### Session Control (Multi-turn Dialogue)

```python
from astrbot.core.utils.session_waiter import session_waiter, SessionController

@session_waiter(timeout=60)
async def waiter(controller: SessionController, event: AstrMessageEvent):
    if event.message_str == "退出":
        controller.stop()
        return
    # process input...
    controller.keep(timeout=60, reset_timeout=True)  # continue session
```

### Calling LLM Services

```python
provider = self.context.get_using_provider()
if provider:
    response = await provider.text_chat("prompt", session_id=event.session_id)
    yield event.plain_result(response.completion_text)
```

**Important:** Always wrap LLM calls with `asyncio.wait_for()` timeout (e.g., 120 seconds) as a safeguard.

**二次审查要求：**

第一次 code-review（常规检查）：
- **Correctness**: Logic errors, inconsistent behavior, edge cases
- **Security**: SQL injection, prompt injection, input validation
- **Performance**: Inefficient operations, missing indexes, N+1 queries
- **Test impact**: Whether existing tests need updates

第二次 code-review（全局逻辑检查）：
在第一次修复完成后，**必须再次运行 code-reviewer 进行全局检查**，不仅检查此次更新引入的问题，还要**全面审查整个代码库的逻辑正确性**，包括但不限于：

1. **数据流完整性**：数据库操作、日志记录、缓存管理是否一致
2. **状态机正确性**：pending_questions 生命周期（创建→作答→标记完成）是否完整
3. **计数一致性**：数据库统计与日志文件计数是否同步
4. **并发安全性**：异步操作是否有适当的锁保护
5. **错误恢复**：异常处理是否覆盖所有可能失败的路径
6. **死代码检测**：是否存在未使用的函数、变量或导入
7. **边界条件**：空输入、超长输入、特殊字符等边界情况处理
8. **业务逻辑**：错题复习优先级、知识点复习排序是否符合设计意图

**Required workflow：**
1. Complete all code modifications
2. Run syntax validation (`python -m py_compile`)
3. **Run code-reviewer agent** (第一次：检查修改引入的问题)
4. Fix any issues reported by code-reviewer
5. **Run code-reviewer agent again** (第二次：全局逻辑检查)
6. Fix any issues reported in second review
7. Only then commit changes

### 上传到 GitHub 时的操作指南

**每次上传代码到 GitHub 后，必须执行以下操作：**

1. **将插件文件复制到 Docker 中的插件目录：**
   ```bash
   # 从 GitHub 克隆/拉取后，将插件文件复制到 Docker 挂载目录
   cp -r /path/to/cloned/repo/* /home/mkconlife/astrbot/data/plugins/review_my_knowledge/
   
   # 或者直接使用 docker cp 命令
   docker cp /path/to/cloned/repo/knowledge_system.py astrbot:/AstrBot/data/plugins/review_my_knowledge/
   docker cp /path/to/cloned/repo/main.py astrbot:/AstrBot/data/plugins/review_my_knowledge/
   # ... 复制其他文件
   ```

2. **将 Init.py 复制到测试目录：**
   ```bash
   # 将 Init.py 复制到外部测试目录
   cp /path/to/cloned/repo/Init.py ../astrbot_plugin_test/
   ```

3. **验证文件同步：**
   ```bash
   # 检查 Docker 容器中的文件
   docker exec astrbot ls -la /AstrBot/data/plugins/review_my_knowledge/
   
   # 检查测试目录
   ls -la ../astrbot_plugin_test/Init.py
   ```

4. **重载插件：**
   - 通过 AstrBot WebUI → 插件管理器 → `...` → `重载插件`
   - 或在 AstrBot 中执行 `/重载复习册` 命令

### 测试注意事项

1. **数据库访问：** 外部测试脚本可以直接访问 `/home/mkconlife/astrbot/data/plugin_data/review_my_knowledge/knowledge.db`
2. **配置同步：** 修改 `settings.txt` 后需要重载插件才能生效
3. **Docker 容器操作：**
   - 查看日志：`docker logs astrbot`
   - 进入容器：`docker exec -it astrbot bash`
   - 重启容器：`docker restart astrbot`
4. **插件重载：** 代码修改后，通过 AstrBot WebUI → 插件管理器 → `...` → `重载插件`
5. **数据导入：** 插件初始化时会自动从 `settings.txt` 配置的 txt 文件导入数据到数据库
